# AISEP (AI-Integrated Software Engineering Process)

<div align="center">
  <p><strong>给 AI 约束，而非指令。让 AI 在约束内自由探索，人类控制边界。</strong></p>
</div>

这是一个将结构化工程约束与大语言模型（LLM）能力深度融合的完整软件工程系统。本 README 基于 **5W2H 框架** 编写，帮助你快速建立对系统的全景认知。

---

## What（是什么？）

**AISEP 不是一个 "AI 自动写代码" 的提词工具，而是一个完整的 AI-人类协作软件工程流水线系统。**
它覆盖了从想法（Idea）到部署复盘（Retrospective）的全生命周期（S0-S8）。系统由五面体架构（控制面/知识面/管理面/进化面/数据面）和 181 个内建方法论/约束文件组成。

## Why（为什么需要它？）

当前 AI 辅助编程仅擅长"单点文件修改"，在面对中大型工程时必然遇到三大死穴：
1. **上下文遗忘**：随着项目推进，AI 忘记早期的架构设计决策。
2. **缺乏系统化领域知识**：AI 写的代码容易偏离框架最佳实践和本地安全规范。
3. **经验无法沉淀**：在这个项目中踩过的坑，在下一个项目里依然会踩。

AISEP的诞生就是为了通过**结构化的约束传递机制（Context Engineering）**和**持久化的自进化记忆架构**解决这三个根本问题。

## Who（由谁使用？谁是主导？）

1. **人类（Human）**：是系统的边界控制者（Quality Gate Keeper）。所有核心决策的推进、规则的更改必须经由人类批准。
2. **AI Agent**：是执行者。在给定的约束和上下文内自主完成代码编写、领域抽象和设计校验。
**合作模式**：Human-in-the-Loop（人在回路）。系统通过 12 条《Constitution 铁律》严格保障人类最终决定权。

## When（何时运作？生命周期是怎样的？）

系统以 **9 阶段流水线（Pipeline）** 驱动。随时可以中途打断、审计、交接。
- **阶段独立（S0-S3）**：初始化 → 领域分析（DDD）→ 需求规格（Stories）→ 架构设计。技术栈无关。
- **循环切片（S4-S6）**：针对每个业务切片（Slice）依次开展：详细设计 → 代码实现 → 测试验证。
- **收尾反思（S7-S8）**：部署配置 → 复盘与系统自进化。

## Where（它在哪里运行、存储什么？）

系统完全基于**本地文件系统（FileSystem as Database）**，结合版本控制（Git）运行。
- **入口围栏**：`AISEP.md`、`constitution.md`
- **系统区**：`.agents/` (工作流+技能), `.aisep/` (文档+配置+模板+进化记忆)
- **项目区**：`projects/{id}/` (纯粹的项目制品与代码)

**核心设计**：更改知识面不影响系统面，更换技术栈只需引入不同的 Framework Skill，实现完美的关注点分离。

## How（系统是如何做到的？）

AISEP 主要依赖 **8 大上下文加载机制** 和 **四层知识注入模型** 保持认知清晰：
1. **三层上下文预算管控**：长驻层（L0, <2K）+ 阶段层（L1, <5K）+ 按需层（L2）。总交互 tokens 永远被压制在 7K-10K 内（次线性增长）。
2. **渐进披露与动态压缩（Compaction）**：过去完成的阶段，其庞大的交付物（如 8KB 的设计文档）在通过 Gate 后会自动压缩为只有百行核心摘要的 `.summary.yaml`。
3. **Skills Gating 门控拦截**：例如系统绝不会在需求分析阶段白白加载如何写 Odoo 视图的知识库。

## How Much（自进化的代价或程度是什么？）

**演化架构（Evolution Architecture）遵循"摩擦力与影响面成正比"**：
- **Level 1（经验记忆）**：基于单次 Gate 审计的修正，**零代价（全自动）** 写入当前项目历史栈。
- **Level 2（知识沉淀）**：复盘提取的框架填坑经验，**低代价（半自动-建议加人审）**晋升为全局可用技能包。
- **Level 3-4（规则重构/能力自生成）**：大范围影响后续所有项目的法则变更，**高代价（强迫人工修改文档）** 必须经由人类直接通过 `/evolve` 干预。
