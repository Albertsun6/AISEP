# ADR-002: 模块策略 — 继承扩展标准模块

## 状态

已接受

## 上下文

确认技术栈为 Odoo 17 后，需要决定自定义模块与 Odoo 标准模块的关系。
Odoo 提供三种继承机制：类继承（扩展原 Model）、原型继承（基于模板创建新 Model）、委托继承（嵌入式组合）。

### 备选方案

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **全量类继承** | 所有自定义模块 `_inherit` 标准 Model | 标准字段/逻辑满足 80%+，仅需扩展 |
| **混合策略** | 核心域新建 Model，支撑域继承 | 核心域需完全控制数据结构 |
| **全量新建** | 不用标准模块，自建所有 Model | 标准模块完全不适用 |

## 决策

采用 **全量类继承** 策略。所有 4 个自定义模块均通过 `_inherit` 扩展 Odoo 标准 Model。

- `mrp_mfg` → 继承 `mrp`（BOM、生产工单、工艺路线）
- `purchase_mfg` → 继承 `purchase`（采购订单），共享 `res.partner`（供应商/客户）
- `stock_mfg` → 继承 `stock`（仓库、库位、出入库）
- `sale_mfg` → 继承 `sale_management`（报价单、销售订单）

## 理由

1. **需求匹配度高** — 分析 40 个 User Stories，标准模块已覆盖 ~85% 字段和 ~70% 业务逻辑
2. **自定义扩展少** — 仅需添加少量自定义字段（如 `qty_variance`、`delivery_status`、`receipt_status`）
3. **复用标准 UI** — 标准模块的 View 可通过 xpath 继承扩展，无需重写
4. **安全复用** — 标准模块已有基础 ACL 和 Record Rules，自定义模块只需补充
5. **数据一致性** — 共用 `res.partner` 实现供应商/客户统一管理，避免数据孤岛

### 核心域为何也用继承？

生产管理虽是核心域，但 Odoo 的 `mrp` 模块已高度成熟：
- BOM 层级管理、工单状态机、原料消耗/成品产出的 stock.move 集成均已实现
- 自定义部分仅为扩展字段（如 `qty_variance`）和增强业务规则（如归档状态）
- 原型继承会导致数据冗余、标准报表不可用，代价过高

## 影响

- **正面**：开发量最小；标准报表/Dashboard 直接可用；升级成本可控
- **负面**：深度定制受限于标准 Model 结构
- **风险**：标准 Model 字段名变更时需同步更新
- **技术债**：须在每个自定义模块中严格通过 `super()` 调用原方法，避免破坏标准流程
